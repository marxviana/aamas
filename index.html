<div id="smells">
    <h2>1. Code Anomalies and Architectural Problems</h2>

    <p>
        Code anomaly, <a
            href="http://eguimaraes15.github.io/index.html#fowler">[1]</a>,
        popularly known as a "code smell", is a micro-structure in the
        program that represents a surface indication of a design problem. In
        this study, we have focus on 10 types of code anomalies (see
        Marinescu <a
            href="http://eguimaraes15.github.io/index.html#marinescu">[2]</a>).
        We present the definition of the code anomalies following:

    </p>
    <div class="questions">
        <blockquote>
            <li><strong> Brain Class. </strong> Complex class that
                accumulates intelligence by brain methods</li>

            <li><strong> Brain Method. </strong> Long and complex method
                that centralizes the intelligence of a class</li>

            <li><strong> Data Class. </strong> Class that contains data but
                not behavior related to the data</li>

            <li><strong> Disperse Coupling. </strong> Method that calls one
                or few methods of several classes</li>

            <li><strong> Feature Envy. </strong> Method that calls more
                methods of single external class that their</li>

            <li><strong> God Class. </strong> Long and complex class that
                centralizes the intelligence of the system</li>

            <li><strong> Intensive Coupling. </strong> Method that calls
                several methods that are implemented in one or few</li>

            <li><strong> Refused Parent Bequest. </strong> Subclass that
                does not use the protected methods of its superclass</li>

            <li><strong> Shotgun Surgery. </strong> Method called by many
                methods that are implemented in different classes</li>

            <li><strong> Tradition Breaker. </strong> Subclass that does
                not specialize the superclass</li>
        </blockquote>
    </div>

    Architecture problems are structures that indicate violation of
    intended design rules or fundamental design principles, and negatively
    impact design quality. In this study we focused on 7 architectural
    problems, described following.

    <p></p>
    <div class="questions">
        <h3>Architecture Erosion</h3>
        <blockquote>
            <li><strong> Ambiguous Interface. </strong> Interface that
                offers only a single, general entry-point but provides two or more
                services.</li>

            <li><strong> Concern Overload. </strong> Component that is
                responsible for realizing two or more unrelated system's concerns.
            </li>

            <li><strong> Connector Envy. </strong> Component that
                encompasses extensive interaction-related functionality that should
                be delegated to a connector.</li>

            <li><strong> Cyclic Dependency. </strong> Components that
                either directly or indirectly depend on each other to function
                properly..</li>

            <li><strong> Scattered Functionality. </strong> Multiple
                components responsible for realizing the same high-level concern,
                with some of them responsible for orthogonal concerns.</li>

            <li><strong> Unused Interface. </strong> Interface that is
                never used by external modules.</li>

        </blockquote>

        <h3>Architecture Drift</h3>
        <blockquote>
            <li><strong> Architectural Violations. </strong> Element or
                relationship that is in the actual architecture but is not in the
                intended architecture, or vice versa.</li>
        </blockquote>

    </div>

    <div id="applications">
        <h2>2. Target Applications</h2>
        <p>In order to conduct this study, we selected 4 target
            applications as described in the following:
        <blockquote>
            <li><strong> Health Watcher </strong> is a web-based
                application that allows citizens to register complaints about
                health issues in public institution.</li>
        </blockquote>

        <blockquote>
            <li><strong> Mobile Media </strong> is software product line to
                derive applications that manipulate photos, videos and music on
                mobile devices.</li>
        </blockquote>

        <blockquote>
            <li><strong> SubscribersDB </strong> is a large software system
                of a publishing house. It manages data related with the subscribers
                of its publications and it supports complex queries on several
                types of data.</li>
        </blockquote>

        <blockquote>
            <li><strong> Apache OODT </strong> is a data-grid system aimed
                at supporting the management and storage of scientific data.</li>
        </blockquote>

        <p>
            These applications were chosen because they met a number of relevant
            criteria. First, they are non-trivial systems and their sizes
            (varying from 10 to 186 KLOC) are manageable for an in-depth
            analysis of code smells. Second, these applications have been
            evaluated in other studies regarding design problems (<a
                href="http://eguimaraes15.github.io/index.html#woizumi">[3]</a>, <a
                href="http://eguimaraes15.github.io/index.html#eguimaraes">[4]</a>,
            <a href="http://eguimaraes15.github.io/index.html#svidal">[5]</a>, <a
                href="http://eguimaraes15.github.io/index.html#tjyoung">[6]</a>, <a
                href="http://eguimaraes15.github.io/index.html#ssoares">[7]</a>,<a
                href="http://eguimaraes15.github.io/index.html#eguimaraes2">[8]</a>).
            Finally, the original developers were available to validate the
            detection strategies for the code smells as well as the
            architectural problems observed for each version.
        <p>Table 1 summarizes the characteristics of the 4 target applications selected

            It is important to say that the more representative versions of the target
            applications were selected, meaning they have a more stable architecture and
            comprises the most important features implemented.
